# Etosha Actual-Data MVP Implementation Plan

## Overview

Build a root-level geospatial pipeline that ingests real-source data for Etosha National Park, normalizes it into analysis-ready layers, computes per-cell threat and species surfaces, and exports a composite risk tensor plus visualizations. The output of this plan is not just a map; it is a reusable spatial foundation for later IMMC work on protection allocation, staffing, and sensitivity analysis.

This plan optimizes for a fast MVP with real inputs. It does not require every source to be fetched automatically on day one. Where a source is public but not realistically scriptable in the available time, the plan allows one-time manual collection and check-in of the resulting artifact, provided provenance is recorded and the transformation into normalized layers is scripted.

## Current State Analysis

The repo is greenfield. At planning time it contains only the rough draft plan, a sparse source list, and the IMMC problem statement:

- [plans/initial_plan.md](/Users/hq/code/immc-risk-intervention/plans/initial_plan.md#L1) sketches a heatmap-oriented approach but mixes assumptions, implementation, and data acquisition in a way that would be hard to verify incrementally.
- [datasources.md](/Users/hq/code/immc-risk-intervention/datasources.md#L1) lists likely external sources but does not yet define file formats, ingestion methods, or provenance tracking.
- [problemstatement.pdf](/Users/hq/code/immc-risk-intervention/problemstatement.pdf) defines a broader objective than visualization alone: the spatial model must support resource allocation, staffing estimation, sensitivity analysis, and adaptation to other parks.

Key constraints discovered from the draft:

- The draft proposes a 5 km grid while working directly in `EPSG:4326` degrees, which is not acceptable for a technically defensible metric grid or distance calculations.
- The draft depends on several external datasets that vary in accessibility: some are likely scriptable, some are webpage-derived, and some may require manual digitization or one-time curated extraction.
- The draft implicitly assumes source data exists in normalized tabular or geospatial form, but the repo currently has no raw or processed artifacts.

## Desired End State

After this plan is implemented, the repo should produce an end-to-end, reproducible-enough MVP from real-source inputs:

- raw source files or documented manually collected artifacts are present under root-level data directories
- all normalized geospatial layers are generated by scripts with consistent CRS and schema
- a metric analysis grid covers Etosha and contains reusable per-cell spatial features
- each threat layer and species layer is derived from actual sourced data rather than placeholder geometry
- a full tensor `R[cell, species, threat]` is saved in a machine-readable format
- static and interactive map outputs are generated for inspection
- validation scripts confirm schema, CRS, geometry validity, expected row counts, and basic score ranges

The plan is complete when a fresh checkout can run the documented pipeline against checked-in inputs and produce the expected outputs, or fail loudly when a required source artifact is missing.

### Key Discoveries

- The IMMC prompt requires a modeling approach for protection over time, staffing, and adaptation, so the spatial risk map must be treated as reusable model input rather than a terminal deliverable.
- The source inventory in [datasources.md](/Users/hq/code/immc-risk-intervention/datasources.md#L1) is incomplete and mixed-quality, so the plan must formalize provenance and ingestion policy before modeling begins.
- The earlier draft’s proposed file structure under `immc/risk_map/` conflicts with the desired root-level implementation target in this repo.

## What We're NOT Doing

- We are not building the full ranger allocation optimizer or staffing simulation in this plan.
- We are not solving the two-other-parks adaptation requirement in code here; we will produce reusable inputs that make that later work easier.
- We are not calibrating every risk weight to a validated ground-truth dataset; where calibration data is absent, we will use explicit literature- or domain-motivated weights and record that limitation.
- We are not requiring fully automated download flows for sources that are realistically only available via map pages, image references, or click-heavy portals.
- We are not building a web app, service, or database unless later work proves that a file-based pipeline is insufficient.

## Implementation Approach

Use a staged geospatial ETL approach at the repo root:

1. define the file contract and provenance rules first
2. ingest or curate actual source artifacts into `data/raw/`
3. normalize every usable source into explicit GIS-ready layers in `data/processed/`
4. compute all metric features against a projected grid in `outputs/`
5. derive risk layers and the species-threat tensor in separate scripts so each stage is independently verifiable

The important architectural choice is to separate:

- source acquisition
- normalization
- feature engineering
- modeling
- visualization

That separation keeps the MVP fast without turning the codebase into a one-off notebook pipeline.

## Phase 3: Metric Grid And Reusable Spatial Features

### Overview

Construct a true metric grid over Etosha and compute reusable per-cell features that all later models will consume.

### Changes Required

#### 1. Choose and enforce a metric CRS
**File**: `scripts/06_build_grid.py`  
**Changes**:

- store source layers in `EPSG:4326`
- project to a metric CRS for grid generation and all distance calculations
- write CRS policy explicitly in code and docs

The exact CRS should be selected based on Etosha’s location in Namibia. The implementation should not silently keep distance work in degrees.

#### 2. Build the 5 km analysis grid
**File**: `scripts/06_build_grid.py`  
**Changes**:

- create 5,000 m x 5,000 m cells in projected coordinates
- clip to park boundary
- assign stable `cell_id`
- export both grid polygons and centroids

#### 3. Compute reusable spatial features
**File**: `scripts/07_build_features.py`  
**Changes**:

- distance to boundary
- distance to fence proxy or park edge
- distance to nearest road
- distance to nearest tourist road
- distance to nearest gate

- distance to nearest waterhole
- pan overlap ratio
- terrain class
- historical fire overlap count or area

```python

grid_metric["dist_to_gate_m"] = grid_metric.geometry.centroid.apply(
    lambda geom: gates_metric.distance(geom).min()
```

## Phase 5: Threat Layers From Real Inputs

### Overview

Generate per-cell threat layers for poaching, wildfire, and tourism using actual infrastructure and event data wherever available, with explicit weighting rules and documented limitations.

### Changes Required

#### 1. Poaching threat surface
**File**: `scripts/09_build_threat_layers.py`  
**Changes**:

- compute a score from actual access and surveillance-relevant features such as gates, roads, boundary proximity, and camps
- incorporate actual rhino-support layer where relevant
- normalize to a bounded score

```python
poaching_score = (
    w_gate * gate_access
    + w_boundary * boundary_access
    + w_road * road_access
    + w_camp * surveillance_gap
    + w_rhino * rhino_support
)
```

The MVP should use explicit, documented weights in code or config. It should not hide defaults or silently infer them.

## Performance Considerations

- GeoJSON is acceptable for small-to-medium vector layers, but intermediate tabular joins should prefer Parquet for speed and schema stability.
- KDE over large point sets can be expensive; cache normalized elephant points and, if needed, pre-bin before smoothing.
- Distance calculations should be vectorized in projected coordinates where possible; avoid repeated per-row geographic computations in degrees.
- The MVP should prioritize correctness and provenance over heavy optimization, but it should avoid designs that force recomputation of every phase after a small downstream change.

## Migration Notes

- There is no legacy implementation to migrate.
- The first stable outputs should be treated as the baseline contract for later optimization and staffing models.
- If a later phase introduces a stronger source for a layer, the processed schema should remain stable so downstream code does not need major rewrites.

## Risks And Mitigations

- Some “actual data” may only be obtainable through manual extraction or terms-limited portals.
  Mitigation: store provenance in the manifest and keep normalization scripted.
- Wildlife sources may be heterogeneous in time period and fidelity.
  Mitigation: record source dates and treat species layers as ecological support surfaces, not synchronized census truth.
- Fire data access may be slower or more complex than expected.
  Mitigation: prioritize a real event-history subset for MVP rather than blocking on perfect temporal coverage.
- Public rhino data may remain intentionally restricted.
  Mitigation: use source-backed habitat and waterhole inference, and state that it is not direct location data.
